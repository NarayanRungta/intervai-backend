1) What This Project Does

The backend continuously reads camera frames, detects face landmarks, estimates eye direction and head pose, checks for mobile phones, computes a suspicion score, and streams the latest snapshot to frontend.

  • Input: webcam frame
  • AI outputs: eye, head, phone, confidence values
  • Final output: MonitoringSnapshot from /status and /ws/monitor

3.2 Per-Frame Processing Flow

flowchart TD
  A[Worker loop tick] --> B[Camera read_frame]
  B -->|fail| E1[Set error camera_read_failed]
  B -->|ok| C[Extract face landmarks]

  C -->|no landmarks| D1[eye/head = unknown]
  C -->|landmarks found| D2{CELERY_ENABLED?}

  D2 -->|No| D3[Run eye + head locally]
  D2 -->|Yes| D4[Queue analyze_landmarks_task]
  D4 --> D5[Wait result timeout]
  D5 -->|task fail| D3

  D3 --> P1[Phone detection with frame skipping]
  D5 -->|task ok| P1
  D1 --> P1

  P1 --> P2{CELERY phone task?}
  P2 -->|No| G[Compute suspicion score]
  P2 -->|Yes| P3[Queue detect_phone_task]
  P3 --> G

  G --> H[Build MonitoringSnapshot]
  H --> I[Store latest_snapshot in memory]

3.3 WebSocket Streaming Flow

sequenceDiagram
  participant F as Frontend
  participant WS as /ws/monitor route
  participant M as MonitoringService

  F->>WS: connect websocket
  WS->>WS: accept + add client in manager
  loop every WS_INTERVAL_MS
    WS->>M: get_latest_snapshot()
    M-->>WS: MonitoringSnapshot
    WS-->>F: send_json(snapshot)
  end
  F->>WS: disconnect
  WS->>WS: remove client

5.2 app/ Entrypoints

app/main.py
  • Creates FastAPI app with CORS.
  • Lifecycle startup creates and starts MonitoringService.
  • Lifecycle shutdown stops monitoring thread and camera safely.
  • Registers /health, /status, /ws/monitor routers.
app/__main__.py
  • Allows running app as module (python -m app).
  • Reads host/port from settings and launches uvicorn.
app/__init__.py
  • Package marker for Python imports.

5.3 app/core/ (Shared App Infrastructure)

app/core/config.py
  • Central settings class loaded from .env using Pydantic Settings.
  • Defines camera, FPS, detection thresholds, model paths, Celery config.
  • Has validators for safe ranges (e.g., websocket interval, celery timeout).
  • get_settings() is cached so config is loaded once.
app/core/logger.py
  • Configures JSON logs for easier debugging in production systems.
  • Provides get_logger(name) helper used by all modules.
app/core/celery_app.py
  • Creates Celery app with Redis broker and result backend.
  • Defines queue name and task serialization settings.
  • Imports monitoring tasks module for worker registration.
app/core/__init__.py
  • Package marker.

5.4 app/api/ (HTTP + WebSocket Layer)

app/api/deps.py
  • Dependency injection helpers to access app-level settings and monitoring_service.
app/api/routes/health.py
  • Implements GET /health.
  • Returns ok or degraded based on worker status and last known error.
app/api/routes/monitoring.py
  • Implements GET /status and WS /ws/monitor.
  • Includes connection manager for multiple websocket clients.
  • Streams snapshots at configured intervals and handles disconnect cleanup.
app/api/__init__.py and app/api/routes/__init__.py
  • Package markers.

5.5 app/services/ (Business + AI Logic)

app/services/camera_service.py
  • Thread-safe wrapper for OpenCV VideoCapture.
  • Opens camera with platform-specific backend flags.
  • Returns (ok, frame) and raises clear startup errors.
app/services/face_landmarks.py
  • Unified landmark extractor with backend fallback:
  • solutions backend (legacy MediaPipe API)
  • tasks backend (modern MediaPipe Tasks API)
  • Returns FaceLandmarksResult(landmarks, error) for robust error reporting.
app/services/eye_tracking.py
  • Uses iris and eye-corner landmark ratios.
  • Classifies gaze into center, left, right, down, unknown.
  • Computes confidence score from how far ratio is from neutral.
app/services/head_pose.py
  • Uses selected face points + OpenCV solvePnP to estimate pose angles.
  • Normalizes yaw/pitch angles to stable range.
  • Maps angles to direction labels and confidence.
app/services/object_detection.py
  • Loads YOLO model and runs inference on each sampled frame.
  • Looks specifically for class id 67 (cell phone).
  • Returns PhoneDetectionResult including errors when inference fails.
app/services/monitoring_tasks.py
  • Celery task definitions:
  • monitoring.analyze_landmarks for eye/head detection
  • monitoring.detect_phone for phone detection
  • Includes helper to build MonitoringSnapshot from task outputs.
app/services/monitoring_service.py (main orchestrator)
  • Runs a background worker loop at target FPS.
  • Reads camera frames and calls landmark extraction + detectors.
  • Supports two modes:
  • CELERY_ENABLED=false => local in-process detection
  • CELERY_ENABLED=true => uses queue tasks with timeout fallback
  • Calculates suspicion score and stores thread-safe latest snapshot.
app/services/__init__.py
  • Package marker.

5.6 app/models/ and app/utils/

app/models/schemas.py

app/utils/helpers.py

7) Topic Deep Dives

Camera Handling

CameraService opens camera lazily and safely with a thread lock. If the camera cannot open, it raises CameraServiceError so monitoring loop can retry and report degraded health.

Face Landmarks

FaceLandmarksExtractor tries solutions first (if allowed), then tasks. This design keeps the system compatible across MediaPipe variations.

Eye Tracking

Uses iris center relative to eye corners to estimate horizontal gaze and iris vertical ratio for down gaze. No deep neural network here; it is geometry-based and lightweight.

Head Pose

Uses solvePnP and fixed 3D face model points to estimate orientation. Then maps yaw/pitch to left/right/up/down/center.

Phone Detection

YOLOv8 inference finds objects and filters for class id 67 or known phone labels. Frame skipping reduces heavy compute cost.

Suspicion Scoring

Score is weighted blend of eye risk, head risk, and phone risk. Weights are configurable in env settings.

7.1 Local Mode vs Celery Mode

ModeSettingProsTradeoffs
Local CELERY_ENABLED=false Simple to run, fewer moving parts All CPU work in API process
Queue CELERY_ENABLED=true Work can scale with separate workers Needs Redis + worker process management

7.2 Error Handling Patterns

  • Camera unavailable: loop retries and health degrades with explicit error.
  • No face in frame: snapshot remains valid but eye/head are unknown with error face_not_detected.
  • YOLO failure: phone detection returns error field instead of crashing worker loop.
  • Celery task timeout/failure: local fallback path runs for landmarks.
